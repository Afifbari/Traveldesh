1. Introduction

1.1 Purpose

The purpose of this document is to provide a detailed overview of our Software Traveldesh, the initial version of which is 1.0. The document will describe about the targeted audience, its user interface, hardware and software requirements. Nonetheless, it will help the designers and developers in the whole software delivery process.

1.2 Document Conventions

This document follows MLA Format. Bold-faced text has been used to emphasize section and sub-section headings. Highlighting is to point out words in the glossary and italicized text is used to label and recognize diagrams.

1.3 Intended Audience and Reading Suggestions

This document is intended towards the users, developers, designers, marketing agents, testers and project managers. This document will assess each audience in grasping the total concept about the app and its features.

1.4 Product Scope

This app is based on traveling purpose. The users can use this app in order to get quick routes towards their destination and also the estimated time and cost for each route that he selects. This can be used by the transportation companies for a better client side user experience in their booking portals.

2. Overall Description

2.1 Product Perspective

The product mentioned in the document will be a solution for the travelers in Bangladesh. The travelers in Bangladesh mostly face difficulties in choosing the right route for their travel and thus are unable to estimate a time and cost for their travelling purpose. This app will assess them to choose the most efficient route and will also give them an idea about the cost and time of travel.

2.2 Product Functions

The major functions of the product are –

  • Let the user choose a departure point.
  • Let the user choose a destination point.
  • Show user all the possible routes based on the departure and destination point selected.
  • Show user the all the possible transportation options for the selected departure and destination point.
  • Show user the estimated time for each route based on the selected transportation system.
  • Show user the estimated cost for each route based on the selected transportation system.

2.3 User Classes and Characteristics

2.3.1 Customer:

Remote customers will mostly use the app for having an assumption about the route their going to travel, they’ll get the idea of the total cost and time of the travel and also the available transportation options. The customers are not expected to be of very technical knowledge and thus the user interface will be made very simple and an easy to understand structure.

2.3.2 Database Administrator:

The Database Administrator is expected to have a college degree and an experience of atleast 2 years in DBA and must have a profound knowledge about IT field. They have the ability to update the informations in the database and manage it accordingly.

2.3.3 Developer:

The developers are expected to have high knowledge about software developing and designing and must have a college degree along with 2 years of software developing experience and also a good knowledge of software related to the product scope. They have to privilege to update and add new features of the app for betterment of user interface.

2.4 Operating Environment

The software will have two platforms in which it will work, one is the website platform and the other is the android app platform. In website platform it will be running with the combination of JavaScript, PHP and Sql. And on the android platform it will be running with the use of Java, PHP and Sql.

2.5 Design and Implementation Constraints

  • Platform requirements: The software must work on android and on website.
  • Time constraint: The software must show the result in less than 4 seconds.
  • Design constraint: The software must have voice recognition system for the user to say their departure and destination points.

2.6 User Documentation

For the user documentation please refer to section 3: External Interface Requirements and attached user manual.

2.7 Assumptions and Dependencies

It is assumed that the software will work fine as long it is used on an android device or on a web browser.
As the software is dependent on feeding latest transportation updates from transportation services through online thus the customer is expected to have internet connection while using the software.

3. External Interface Requirements

3.1 User Interfaces

The user interface of Traveldesh has been designed in such a way so that the user doesn’t have to roam around to find options and get annoyed. The interface is easy to use and thus the user will get the hold of the software within seconds.

The home screen will have the option for user to choose destination and then after that will have the option to choose the departure point and based on that resulted routes, distance and transportation options will appear.

After selecting the transportation option the user will get the estimated cost and time in the travel based on that transport. The user will also have the option to switch between the transportation options and compare the time and cost for each case.

3.2 Hardware Interfaces

The software will have the capability to run on computers and as well as android device. In computer the software will run as a website application which will be accessible with a browser and on the android device the software will run as android app as well as a web application which will be again accessible by a mobile browser.

3.3 Software Interfaces

The software runs on Java based android app as well as JavaScript based web app. In both cases the software has the ability to recognize the user voice in terms of selecting their departure and destination points.

3.4 Communications Interfaces

Users need to connect the software to the main system by the means of internet. From there they can easily perform their tasking selecting and searching for their travel routes. They will be able to login to the app with their registered email address and password. Whenever the user connects the software to the internet the software will automatically take the required updates. The update will completed within a very short period of time.

4. System Features

4.1 Use case name – Take Departure Point

	4.1.1 Objective – The purpose of this case is to take departure point from the user through the means of voice recognition.

	4.1.2 Source – The source of this case is the user.

	4.1.3 Actors – The end user of the device which is the customer would be the actor of this case.

	4.1.4 Flow of events –

		4.1.4.1 Basic Flow –  

			4.1.4.1.1 User says the departure point.
			4.1.4.1.2 Software recognizes the point and goes to next feature.

		4.1.4.2 Alternative Flow – At step 4.1.4.1.1, the user decides to type the departure point.
			4.1.4.2.1 User presses cancel button.
			4.1.4.2.2 Software shows text field to enter point.
			4.1.4.2.3 User types point and presses enter.
			4.1.4.2.4 Software takes the chosen point and goes to next feature.

		4.1.4.3 Exception Flow – At step 4.1.4.2, the device can’t recognize the point.

			4.1.4.3.1 – The software shows an error message and let’s the user to again say the departure point.

	4.1.5 Post conditions – The user is taken to the destination selection feature.

4.2 Use case name – Take destination point

	4.2.1 Objective – The purpose of this case is to take destination point from the user through the means of voice recognition.

	4.2.2 Source – The source of this case is the user.

	4.2.3 Actors – The end user of the device which is the customer would be the actor of this case.

	4.2.4 Flow of events –

		4.2.4.1 Basic Flow –  

			4.2.4.1.1 User says the destination point.
			4.2.4.1.2 Software recognizes the point and goes to next feature.

		4.2.4.2 Alternative Flow – At step 4.2.4.1.1, the user decides to type the destination point.
			4.2.4.2.1 User presses cancel button.
			4.2.4.2.2 Software shows text field to enter point.
			4.2.4.2.3 User types point and presses enter.
			4.2.4.2.4 Software takes the chosen point and goes to next feature.

		4.2.4.3 Alternative Flow – At step 4.2.4.1.1, the user decides to go back and give the departure point again.
			4.2.4.3.1 User presses back button.
			4.2.4.3.2 Software takes the user back to the departure selection menu.


		4.2.4.4 Exception Flow – At step 4.2.4.2, the device can’t recognize the point.

			4.2.4.3.1 – The software shows an error message and let’s the user to again say the destination point.


	4.2.5 Post conditions – The user is taken to the select transportation option feature.

4.3 Use case name – Select transportation option

	4.3.1 Objective – The purpose of this case is to take transportation option from the user through the means of selections from a set of available transportations. 

	4.3.2 Source – The source of this case is the user.

	4.3.3 Actors – The end user of the device which is the customer would be the actor of this case.

	4.3.4 Flow of events –

		4.3.4.1 Basic Flow –  

			4.3.4.1.1 User selects the transportation option.
			4.3.4.1.2 Software takes the option and shows the estimated time and cost for that selected transportation option.

		4.3.4.2 Alternative Flow – At step 4.3.4.1.2, the user decides to view a different transportation option
			4.3.4.2.1 User presses another transportation option button.
			4.3.4.2.2 Software goes back to step 4.3.4.1.2


		4.3.4.3 Exception Flow – At step 4.3.4.1.2, the user decides to go back to home.

			4.3.4.3.1 – The user presses home button.
			4.3.4.3.2 – The user is taken back to take departure point menu.

5. Other Nonfunctional Requirements

5.1 Performance Requirements

The software must work seamlessly while its operation. The filtered options must be shown within a time limit of 4 seconds and the software must not crash during the operations. The crash rate must be less than 2 percent.

5.2 Software Quality Attributes

The voice recognition system used by the app must recognize the app precisely and the success rate must be at least 98 percent. The app must renew it’s database time to time so that the informations fetch to the users are accurate and reliable. The user interface must be such that even the person with the lowest technical knowledge can use the app with least difficulties.

6. Other Requirements

For information regarding functional requirements, refer to section 4: System Features.
Each feature has its requirements listed alongside the feature information. Special requirements for each system feature are also mention in section 4. There are no additional Functional requirements.
